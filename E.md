# Slide 13: Complexity — Methods of Handling

- Header: **2. Complexity**
- Subtitle: **Methods of Handling**
- Core Matrix (Cognitive Tools for Simplification):
  - **Abstraction?** -> *Using Equivalence Relations Omitting Details with Focus on the Essential.*
  - **Classification?** -> *Assign Entities to Groups (narrow/more formal).*
  - **Categorisation?** -> *Assign Entities to Groups (broad/more mental).*
  - **Reduction?** -> *Omitting Details without String Focus on the Essential.*

### Conceptual Explanation

Having established that complexity is a cognitive burden (Slide 12), this slide defines the four primary mental tools—the "Simplifiers"—that an intelligent system uses to handle it. 

These are the operational algorithms the mind uses to compress data:
1. **Abstraction**: The most rigorous tool. It actively searches for "equivalence relations" (shared mathematical or structural properties) across different entities, stripping away noise until only the core, essential mechanism remains.
2. **Classification**: A formal, hard-coded sorting mechanism. Entities are placed into rigid, narrow groups based on strict, predefined rules (e.g., biological taxonomy).
3. **Categorisation**: A softer, more flexible sorting mechanism. Entities are placed into broad, subjective groups based on mental utility or fuzzy boundaries (e.g., "things you take on a picnic").
4. **Reduction**: The bluntest tool. It simply deletes or ignores data to save space, but unlike abstraction, it doesn't guarantee that the remaining data is the "essential" part. It is a lossy compression.

### Why this matters

- For computer scientists building AGI or machine learning models, this slide is essentially a catalog of data processing techniques. 
- A neural network uses **Categorisation** (clustering algorithms) to find fuzzy patterns. A database uses **Classification** (rigid schemas). A physics simulation uses **Abstraction** (ignoring friction to calculate baseline gravity). A downsampling algorithm uses **Reduction** (dropping pixels to reduce file size). Understanding which tool to use is the key to building efficient, non-crashing architectures.

### Interpretive note

- The typo *"String Focus"* (likely meant to be *Strict Focus*) in the Reduction box highlights the difference between purposeful abstraction and blind reduction. This distinction is vital in AI safety and alignment—if an AI uses blind reduction instead of careful abstraction, it might delete critical safety parameters.

# Slide 17: Information Theory — Thesis

- Header: **3. Information Theory**
- Subtitle: **Thesis**
- Core Definitions:
  - **What is information?**
    - Information is a mental entity in the mental world.
    - Information is always part of a brain or thought machine (which itself is part of the physical world – body-mind-dualism).
    - Information is always representing other entities.
    - Information can have tree granularities (atomic, elementary, complex).
    - Information processing (logic) and persisting (memory) is the task of a brain.
  - **What is an entity?**
    - An entity is everything that is delimitable: *No delimitability, no entity*.
    - This applies in both the physical world and the mental world.

### Conceptual Explanation

This slide establishes the foundational thesis statements that anchor the presenter's specific flavor of Information Theory. 

First, it strictly defines **Information** as a subjective, mental phenomenon. It exists only inside a "brain or thought machine" (a processor). However, it acknowledges the "body-mind dualism" of computing: while the information itself is abstract and mental, the hardware running it (the brain/CPU) is a physical object. Furthermore, information is fundamentally a pointer—it always represents or maps to something else. It also introduces the idea that information is hierarchical, scaling from atomic bits to complex schemas.

Second, it provides a universal definition for an **Entity**. An entity is defined purely by "delimitability" (boundaries). If you cannot draw a boundary around something to separate it from its environment, it cannot be considered an entity. This rule applies equally to a physical object (like a rock with physical edges) and a mental object (like a specific memory or an isolated variable in code).

### Why this matters

- For computer scientists, this defines the strict rules of object-oriented programming and data architecture. 
- "No delimitability, no entity" means that an AI cannot process data if that data isn't bounded. A variable must have a defined type and memory allocation; an object in computer vision must have a bounding box. If the AI cannot delimit the data, it cannot turn it into an information entity.

### Interpretive note

- The typo *"tree granularities"* instead of *three granularities* is preserved from the original slide. The specific tiers (atomic, elementary, complex) will likely be expanded upon in subsequent slides as the basis for data structures.

# Slide 34: Knowledge Gap — Concept Schemas

- Header: **6. Knowledge Gap**
- Subtitle: **Concept Schemas**
- Core Concept:
  - Through our whole lives we learn a lot of concept schemas.
  - Example: Concept "father" has the schema "father has one or more childs" (meaning one or more associations or elementary information units).
- Execution Loop:
  - Once we hear that a person (Robert) is a father, we immediately:
    1. Think of the imprinted/learned concept schema "father."
    2. Detect the information gap (the missing atomic child information).
    3. Are used to ask for the number of children and for the children.
  - We ask automatically to fill the knowledge gap. 
  - The answer fills the gap: "He has three childs."
- Visual Flow:
  1. **Concept Schema**: Abstract template (`father` $\rightarrow$ `child 1 ... child n`).
  2. **Concept instance (uncomplete network)**: The template applied to a real entity with missing data (`Robert` $\rightarrow$ `???`).
  3. **Complete network (without names)**: The gap is filled (`Robert` $\rightarrow$ `child 1`, `child 2`, `child 3`).

### Conceptual Explanation

This slide answers the workshop prompt from Slide 33 by explaining the mechanics of "Concept Schemas." 

A concept schema is essentially an empty data template. Over time, an intelligent agent learns that certain entities always have specific attributes. If you learn the concept of a "car," the schema requires it to have "wheels" and an "engine." 

When the agent encounters a new entity (like "Robert is a father"), it instantly pulls up the "father" template. The template has blank slots for "children." Because the slots are empty, the agent algorithmically detects an **Information Gap**. The detection of this empty slot is what triggers **Motivation**—the agent is driven to act (ask a question) to fill the empty slots and complete the network.

### Why this matters

For computer scientists, this slide describes **Schema Instantiation** and **Slot Filling** in Natural Language Processing (NLP) and AI design.
- If you build an AI assistant to book flights, you give it a "Flight Schema" with slots for `[Destination]`, `[Date]`, and `[Passenger Name]`.
- If a user says, "Book a flight to Paris," the AI instantiates the schema. It fills `[Destination]` but detects gaps for `[Date]` and `[Name]`. 
- This structural gap *motivates* the AI to ask the user, "For what date?" 

This is how you program curiosity and motivation into an AGI: you give it templates of the world and compel it to fill in the blanks.
